System narrative, not spaghetti

This page is intended to explain Jarvis the way a strong CTO architecture review would: in a clean narrative arc. Inputs enter the system, orchestration converts intent into governed execution, memory and retrieval ground the work, outputs are delivered through real operating channels, and recovery is handled through separate backup domains.

Recovery and redundancy

Recovery sits underneath the operating flow because it protects the system without confusing the narrative. Backup strategy is intentionally separate by domain: vault recovery is not the same as workspace recovery.

domain-specific backups vault strategy still open
Knowledge recovery

Vault Local Mirror

Local Mac backup of the SSD vault. Strong second copy, but third-copy standardization is still unresolved.

Execution recovery

Workspace Local Mirror

Local backup of the active OpenClaw workspace for fast restore on the Mac mini.

Remote recovery

GitHub Versioned Backup

Remote, versioned recovery layer for the workspace only, not the full vault.

What a CTO should take away open

The system is strongest when read as a layered operating model, not as a visual effects page.

  • The core boundary is knowledge source of truth versus live execution environment.
  • Integrations matter because they connect the machine to real work, not because they make the stack look broader.
  • Backups should be discussed by domain so recovery logic stays legible.
