System thesis

How to read Jarvis

  • Jarvis is not a single folder or model; it is an operating system across runtime, memory, retrieval, integrations, and outputs.
  • The architectural center of gravity is the distinction between knowledge source of truth and live operating environment.
  • If that distinction is respected, backup strategy, retrieval design, and interface design all become easier to reason about.

Source-of-truth matrix

Memory
Obsidian vault on external SSD is the primary memory / knowledge source of truth.
Runtime
OpenClaw workspace on local disk is the live operating environment for Jarvis.
Workspace backup
Local Mac mirror plus GitHub versioning.
Vault backup
Local Mac mirror exists; third-copy strategy remains the open architecture decision.

Integration surfaces

Google Workspace

  • Calendar powers scheduling + morning brief inputs
  • Gmail supports read / draft / send flows
  • Drive / Docs / Sheets act as operational output surfaces
  • Jarvis Drive folder is the default parent for shareable assets

Operator channels

  • Telegram / chat interfaces for direct execution
  • Briefs, reminders, and updates delivered through messaging surfaces
  • Human review remains central for external and higher-risk actions

Critical architecture rules

Do not confuse these layers

  • Vault is not just a backup of workspace
  • Workspace is not just an implementation detail of the vault
  • Generated indexes are not strategy docs
  • Backups should be explicit by domain, not implied by habit
